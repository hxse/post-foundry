# 00 Meta

## 任务概括

建立 PostFoundry 第一主任务的正式设计快照。PostFoundry 的长期愿景是让 Codex 中的 AI 成为少量自有 X 账号的长期运营助手：统计热点、查找和核验资料、形成选题、生成草稿、排程发布、读取表现并复盘经验。这个 root task 不实现完整产品，只冻结项目起步阶段的技术栈、模块边界、多账号隔离原则、外部 API 使用边界、验证入口和后续子任务拆分，确保后续实现不偏离这个运营闭环。

## 正式 task 级别及定级原因

三星任务。原因是本任务会决定项目的长期基础契约：多账号隔离、账号身份映射、配置与密钥边界、真实发帖保护、外部 API 读写分层、`just` 验证入口和后续 task 的拆分方式。如果这些边界一开始不清楚，后续很容易出现账号串号、真实发帖误触发、配置漂移、在线测试混入默认 Gate 或运营经验互相污染等 quietly wrong 风险。

## 范围内与范围外

范围内：

* 冻结 PostFoundry 第一阶段的项目定位和非目标。
* 冻结长期产品愿景：AI 辅助自有账号做选题、资料研究、草稿、排程发帖和复盘，不做水军、刷量或批量操控。
* 冻结 TypeScript、Bun、克制使用的 SvelteKit、Vite、SQLite、Drizzle、Zod、Vitest 和 `just` 的基础技术选型。
* 冻结多账号 by design 的核心原则：系统支持 N 个账号，所有业务数据以内部不可变 `account_uuid` 隔离，对外可配置 `account_key` 可重命名。
* 冻结 X 数据读取和账号动作的边界：第三方数据 API 只读公开数据；X 官方 API 做发帖等账号动作；不做网页登录自动化、反检测浏览器、cookie 代理或自动控评。
* 冻结第一主任务的子任务拆分，并详细展开第一个 API 连通性子任务。
* 冻结离线测试与在线 debug 的验证口径：在线、真实外部服务和可能计费的验证只能由用户明确要求后手动运行，不进入默认测试、CI 或 agent 自主 Close Gate。

范围外：

* 不实现代码、不安装依赖、不创建运行时数据库。
* 不实现完整 OAuth 授权 UI。
* 不实现正式选题系统、LLM 草稿生成、排程发帖队列、复盘系统、评论审核后台或可视化报表。
* 不接入真实 X 账号，不执行真实发帖，不写任何 API key 或 token。
* 不创建反检测浏览器、网页登录自动化、模拟真人轨迹、代理池、cookie 托管或绕过平台风控的能力。
* 不把在线、真实外部服务或可能计费的 API 调用做成自动测试。
