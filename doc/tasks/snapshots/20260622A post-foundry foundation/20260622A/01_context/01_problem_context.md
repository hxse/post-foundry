# 第一主任务背景

PostFoundry 要解决的是个人或小团队运营 X 账号时的重复劳动：选题、资料查找、草稿、排程、数据回看、经验沉淀和评论审核辅助。项目初期只跑一个账号，但架构不能写死一个或三个账号，因为后续新增账号应该只是新增配置和授权，不应该改代码。

长期愿景是让 Codex 中的 AI 成为自有 X 账号的长期运营助手。它应该能持续观察公开热点，读取第三方公开数据，查找和核验资料，按账号定位形成选题，写出自然但可控的草稿，按节奏定时发帖，读取公开表现数据，并把哪些主题、表达方式和发布时间有效沉淀为账号级记忆。

这个愿景不是做水军、刷量或批量账号操控。PostFoundry 不追求伪装成一群人，也不追求自动互动放大声量；它追求的是让一个真实拥有者的账号长期稳定产出有资料、有判断、有风格的内容。系统可以越来越自动化，但每一步都必须可审计、可配置、可回放，并且不能绕过平台规则。

这个项目的风险不在页面复杂度，而在边界。账号不能串，内容策略不能混，API 测试不能误发真实帖子，在线服务不能混进默认测试，运营经验不能从一个账号污染到另一个账号。如果这些边界靠约定记忆，后面一旦加账号、加调度、加 LLM 或放进 Podman 长期运行，就会出现难排查的错误。

## 使用场景

用户在本地运行一个轻量运营后台。系统定期从公开数据源和第三方 X 数据 API 读取公开信息，统计热点，查找补充资料，按每个账号自己的主题、语言、风格和禁区生成候选选题与草稿。运营者审核或配置允许后，系统通过 X 官方 API 定时发帖。发帖后的表现、主题、时间、表达方式和复盘结论沉淀到该账号自己的记忆里。

第一阶段的目标不是把完整运营闭环一次做完，而是先把项目骨架和 API 真值链做稳。这条路线看起来离最终自动运营较远，但它是在给后续闭环铺地基：账号不能串、API 不能误用、真实发帖不能误触发、在线费用不能被自动测试消耗、历史表现必须能归因到正确账号。特别是 API 相关能力必须先被离线 fixture 证明；在线 debug 只能作为用户明确要求后的人工诊断，后续才值得在它之上实现选题、发帖队列和复盘。

## 需求痛点

第一，热点扫描和公开 X 数据读取不走 X 官方 API。当前方案使用第三方 X 数据 API 读取公开数据，例如 TwitterAPI.io；X 官方 API 只负责官方允许的账号动作，例如发帖。成本风险主要来自第三方数据 API 的查询次数、返回条数和后续 LLM 调用；X 官方 API 在第一阶段的风险主要是发帖权限、rate limit、OAuth 授权和真实副作用。这个读写分层必须在项目第一天就写进 contract，避免后续把热点扫描误接到 X 官方 API，或把发帖动作误接到第三方数据 API。

第二，多账号不是简单多几行配置。每个账号有自己的语言、主题、风格、预算、数据源、草稿队列、发布节奏、历史指标和经验复盘。系统可以共享底层 provider 和通用工具，但内容决策必须进入 account-specific pipeline。否则中文技术号、英文技术号和政治新闻号的素材、风格或经验会互相污染。

第三，SvelteKit 适合做本地管理台，但不能让业务逻辑被页面和 route 吞掉。PostFoundry 的核心业务应该是普通 TypeScript 模块，SvelteKit 只做薄 UI 壳、少量 server endpoints、OAuth callback 和表单状态展示。这样后续即使换 UI 或加 CLI，核心逻辑也能复用。

第四，在线 API 测试有真实副作用，也可能产生第三方 API 费用。X 发帖、隐藏回复、OAuth token 验证、TwitterAPI.io 查询和第三方读回都不能混进默认测试。默认 Gate 必须离线、快速、稳定、可重复；在线能力只能在用户当前明确要求时通过明确的 `just debug...` 命令手动运行，真实发帖还必须再加硬开关。

## 候选方案与取舍

后端语言有 Python 和 TypeScript 两个候选。Python 在 LLM、数据分析和脚本生态上有优势，但 PostFoundry 的主线是长期运行的本地后台、API 编排、多账号配置、OAuth、任务调度和 UI 管理台。TypeScript 能把前端、后端、配置 schema、API client 和测试放在同一类型体系里，因此第一阶段选择 TypeScript。后续如果复盘分析需要 embedding、聚类或更重的数据实验，可以单独增加 Python worker，但不进入第一阶段。

前端和服务端有 raw Vite 加独立后端、SvelteKit 全栈、Next.js 三个候选。raw Vite 只适合纯前端，OAuth callback、本地 API、SQLite 和 debug endpoint 还要另配 Hono 或 Fastify；Next.js 对这个本地轻量工具偏重。SvelteKit 已经基于 Vite，能提供页面、server endpoints 和本地管理台需要的最小后端能力。取舍是克制使用 SvelteKit，不把它当复杂全栈框架，不把业务逻辑写进 route。

数据库有 Postgres、Redis 加队列、SQLite 三个候选。这个项目初期不重，主要本地运行，后期 Podman 单容器部署。Postgres 和 Redis 会引入额外服务和运维成本，第一阶段不需要。SQLite 文件数据库足够承载账号配置、草稿、发帖记录、API 调用日志、预算记录和复盘记忆。后续如果并发和任务量真的变大，再迁移到 Postgres。

API 验证可以拆成离线测试子任务和在线测试子任务，也可以放在同一个子任务中。当前选择放在同一个 `20260622A.001 api connectivity harness` 子任务里，因为二者验证的是同一个 API contract。离线测试是正式 Gate，在线测试是用户明确要求后的手动 debug，不得自动化运行；它们应该共享同一套 client、schema、错误语义和 fixture 设计，而不是拆成两个可能漂移的任务。

## 取舍结论

第一主任务先做基础，不做完整 MVP。它拆成三段：先做 API 连通性验证，再做多账号配置隔离，再做运行骨架和存储基线。选题、草稿、发布队列、复盘和评论审核作为后续 root task 承接。

技术栈固定为 Bun、TypeScript、克制使用的 SvelteKit、Vite、SQLite、Drizzle、Zod、Vitest 和 `just`。系统从第一天支持 N 个账号，但账号数量是运营和预算限制，不是代码结构限制。所有账号相关业务表必须以内部不可变 `account_uuid` 隔离，对外配置用可变 `account_key`。
